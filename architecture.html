2. Database choice — why Supabase

Recommended

Supabase

  • Postgres — real SQL, foreign keys, JSONB for pour schedules
  • Auth included — magic-link, zero code
  • Realtime — friends feed updates live
  • Row Level Security — DB-level permission enforcement
  • Generous free tier, not locked-in (just Postgres)
Alt

Turso (SQLite)

  • Cheap, fast reads
  • Need to add auth + permissions ourselves
  • No realtime
Alt

Firebase

  • Similar features
  • Schema-less, vendor lock-in
  • Unpredictable billing

Helper
Grind translation (xBloom → universal)
Pure function in code — no DB column needed (derived on the fly):
function xbloomToDescription(g: number): string {
  if (g <= 48) return 'Fine'
  if (g <= 54) return 'Medium-fine'
  if (g <= 60) return 'Medium'
  if (g <= 68) return 'Medium-coarse'
  return 'Coarse'
}

export async function saveBrew(input: unknown) {
  const data = BrewSchema.parse(input)           // Zod — again, trust nothing
  const { user } = await supabase.auth.getUser() // RLS uses this
  const bean = await findOrCreateBean(data.bean, data.roaster, data.origin)
  await supabase.from('brews').insert({
    user_id: user.id, bean_id: bean.id,
    brew_method: data.method, drink_style: data.drink,
    dose_g: data.dose, water_ml: data.water,
    ratio: data.water / data.dose,

    grind_description: xbloomToDescription(data.grind),
    rpm: data.rpm, pours: data.pours,
    rating: data.rating, notes: data.notes,
    visibility: 'friends'
  })
}

5. Security — 4 layers

Layer 1

Auth

Supabase magic-link. No passwords on our end. Sessions refresh via secure cookies. Only verified email can access.

Layer 2

Row Level Security

Postgres enforces permissions at the DB. Even if a bug sends a wrong query, the database rejects it.

-- brews: you can only touch your own rows
CREATE POLICY "brews_own" ON brews
  FOR ALL USING (user_id = auth.uid());

-- brews: friends can read your friends-visible brews
CREATE POLICY "brews_friends_read" ON brews
  FOR SELECT USING (
    visibility = 'friends'
    AND EXISTS (
      SELECT 1 FROM friendships
      WHERE user_id = auth.uid() AND friend_id = brews.user_id
      AND status = 'accepted'
    )
  );
Layer 3

Input validation

Every Server Action runs Zod before touching the DB. Bad payloads fail fast with a clear error.

const BrewSchema = z.object({
  bean: z.string().min(1).max(120),
  method: z.enum(['xBloom','V60','Chemex','AeroPress','Espresso','French press','Cold brew']),
  dose_g: z.number().positive().max(100),
  water_ml: z.number().int().positive().max(2000).optional(),
  water_temp_c: z.number().min(60).max(100),
  grind_xbloom: z.number().min(1).max(100).optional(),
  rating: z.number().int().min(1).max(5),
  notes: z.string().max(2000).optional(),
  pours: z.array(z.object({
    label: z.string(), ml: z.number().int().positive()
  })).optional()
})
Layer 4

Secrets & transport

  • Supabase keys in .env.local, gitignored
  • HTTPS everywhere via Vercel
  • Rate limiting on auth endpoints (Supabase default)
  • Service-role key never sent to browser

Pre-deploy checklist
  • All RLS policies tested with a second user account (verify you can't read others' private brews)
  • Zod schema on every Server Action
  • Unit test coverage on business logic (ratio, grind translation, autofill)
  • E2E test passing for the critical path
  • Error monitoring set up (Sentry free tier or Vercel logs)
  • Data export tested end-to-end
